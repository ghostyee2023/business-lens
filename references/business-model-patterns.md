# Business Model Patterns

Pick the smallest model that can test willingness to pay.

Choose the model after buyer, value capture, acquisition, and delivery are explicit. A model name is not a commercial path by itself.

## Lightweight Models

- Paid template: sell a reusable artifact that solves a narrow repeated job.
- Service-assisted MVP: deliver manually first, automate later.
- Workshop: teach and apply the method live for a defined audience.
- Diagnostic report: analyze an input and produce a decision-ready output.
- One-off tool: charge for a specific transformation or utility.
- Subscription: use only when the job recurs and fresh value appears.
- Plugin: monetize inside an existing workflow or platform.
- Lead generation: free tool creates qualified demand for a higher-ticket service.
- Internal productivity product: sell to teams when time saved maps to budget.
- Data product: sell structured, maintained information that is painful to gather.

## Model Fit

| If the value is... | Start with... |
| --- | --- |
| expert judgment | diagnostic report or workshop |
| repeatable output | template or one-off tool |
| ongoing monitoring | subscription |
| workflow convenience | plugin |
| trust-heavy service | service-assisted MVP |
| audience education | paid guide, cohort, or workshop |
| proprietary organization | data product or knowledge base |

## Avoid Premature SaaS

Do not default to SaaS when:

- the user does not know the buyer
- delivery can be tested manually
- the workflow is not repeated
- onboarding requires too much trust
- the first 10 users are not reachable

SaaS is often the second business model, not the first validation.

## Pricing Hypothesis

State a testable price, range, or pricing basis. Tie it to one of:

- replacement spend
- value created or risk avoided
- current labor cost
- comparable procurement category
- paid pilot scope

Avoid "freemium then subscription" without a reason the buyer will cross the payment boundary. The first price test should ask for real money or a purchasing commitment from the proposed buyer.

## Delivery Reality Check

For the selected model, state:

- who performs the work per customer
- what input the customer must provide
- time to first value
- expected human effort per delivery
- what becomes reusable after each customer

If effort scales linearly, price it as a service until repeatability is demonstrated.
